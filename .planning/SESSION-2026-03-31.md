## Current State: NOT FULLY WORKING

### Blocking Issue: Can't Get Into The App
The device picker screen is showing "Finding devices..." or "Spotify rate limit hit" for new sessions. Root cause chain:
1. Early version of auto-retry in DevicePicker looped every 5s, hammering Spotify
2. Spotify responded with a long `Retry-After` header
3. Our retry logic was honoring it — sleeping 60s × 3 retries = 3+ minute silent hang
4. This created an unrecoverable loop that got the app's Spotify client rate-limited

**Mitigation shipped:** Device fetched from localStorage (no API call) if previously selected. But first-time users or signed-out users still hit this.

**Rate limit will clear on its own** — Spotify rate limits are typically per-minute or per-hour windows. After ~1 hour of no requests it should be fine.

## Known Issues / Debt

### High Priority (Blocking Usability)
1. **Device picker reliability** — `getDevices()` Spotify call can hang or rate-limit. Need a more robust strategy: maybe skip device picker entirely (rely on Spotify's active device via playback state polling instead of explicit device selection)
2. **Rate limit handling overall** — `withErrorHandling` retry logic is too aggressive. Should fail fast (1 retry max, short sleep) for all non-critical calls like device fetch
3. **No local dev testing** — Spotify redirect URI is only configured for `tunes.dylandibona.com`. Can't test at localhost. Every iteration requires a Vercel deploy + live test. Slow and risky.

## Architecture Notes (For Next Claude)

The app architecture is sound. The bugs this session were all at the integration layer:
- Spotify SDK error format != what our error handler expected
- Retry logic too aggressive (slept too long, looped too eagerly)
- Polling started too early (before device selection, causing rate limits)
- Device selection not persisted (caused repeated API calls on every load)

The core vibe → recommendation → playback pipeline is structurally correct. The issues are operational/reliability, not architectural.
